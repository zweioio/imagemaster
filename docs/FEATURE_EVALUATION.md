# ImageMaster Pro 功能评估报告

## 1. 视频转 GIF 功能评估

### 1.1 核心需求
将用户上传的视频文件（MP4, WebM, MOV 等）转换为 GIF 动图，支持设置帧率、尺寸、画质和截取时间段。

### 1.2 技术方案
由于项目已集成 `gif.js` 和 `gifsicle-wasm-browser`，我们采用 **原生 Video + Canvas 方案**，充分利用现有基础设施，避免引入 `ffmpeg.wasm` (20MB+) 造成的体积膨胀。

**处理流程：**
1.  **解码**：使用浏览器原生 `<video>` 标签加载视频。
2.  **抽帧**：使用 `requestVideoFrameCallback` 或 `timeupdate` 事件，将视频帧绘制到 `OffscreenCanvas`。
3.  **编码**：复用现有的 `gif.js` 将 Canvas 帧序列编码为 GIF。
4.  **压缩**：复用现有的 `gifsicle-wasm-browser` 对生成的 GIF 进行二次压缩优化。

### 1.3 难度与工作量评估
*   **难度等级**：⭐⭐⭐ (中等)
*   **预计工时**：2-3 天
*   **主要难点**：
    *   **内存管理**：视频抽帧会产生大量 ImageData，容易导致浏览器崩溃 (OOM)，需要实现分块处理或流式处理。
    *   **UI 交互**：需要开发时间轴滑块（Range Slider）来选择视频的开始和结束时间。
    *   **性能**：在主线程抽帧会卡顿 UI，需将抽帧逻辑移至 Web Worker (Sandbox)。

### 1.4 优缺点
*   **优点**：无需额外下载大文件；复用现有压缩逻辑；符合“本地处理”隐私原则。
*   **缺点**：不支持复杂的视频滤镜（如调色），仅支持基础剪辑。

## 2. 图片去水印 (AI Inpainting) 功能评估

### 2.1 核心需求
用户涂抹图片上的水印或多余物体，AI 自动填充背景，使其消失。

### 2.2 技术方案
采用 **ONNX Runtime Web + LaMa (Large Mask Inpainting) 模型**。LaMa 是目前修复效果最好且速度较快的开源模型之一。

**处理流程：**
1.  **交互**：Canvas 画笔涂抹，生成蒙版 (Mask) 图片（黑底白画）。
2.  **推理**：将原图和蒙版输入 ONNX 模型。
3.  **合成**：模型输出修复后的图像，在前端展示。

### 2.3 模型选择
*   **模型**：LaMa-Regular (量化版) 或 ZITS。
*   **大小**：
    *   原始模型：~200MB (过大，不推荐)。
    *   **量化模型 (Int8/FP16)**：约 **30MB - 50MB**。适合本地插件集成。
*   **性能**：M1 芯片支持 WebGPU 加速，推理时间预计在 1-3秒。

### 2.4 难度与工作量评估
*   **难度等级**：⭐⭐⭐⭐⭐ (高)
*   **预计工时**：4-5 天
*   **主要难点**：
    *   **Canvas 交互**：需要实现高性能的画笔涂抹、撤销/重做、笔刷大小调节功能（类似 Photoshop 的简单版）。
    *   **模型集成**：LaMa 模型的输入输出预处理（Resize, Padding）比抠图模型复杂，且需要寻找或转换合适的 ONNX 模型文件。
    *   **包体积**：即使量化后，插件体积也会增加 30MB+，可能影响分发。

### 2.5 优缺点
*   **优点**：本地运行，隐私安全；效果惊艳，增加产品竞争力。
*   **缺点**：开发成本高；模型文件大；对低端设备（非 GPU）性能压力大。

---

## 3. 综合建议

**建议优先开发：视频转 GIF**

1.  **性价比高**：利用现有技术栈即可实现，无需引入新模型，开发风险低。
2.  **用户刚需**：视频转 GIF 是常见的办公/媒体需求，与“图片处理”定位契合。
3.  **技术验证**：可以作为 Canvas 性能优化的练兵场，为后续去水印的 Canvas 交互打基础。

**去水印功能**建议作为 **2.0 版本** 的核心亮点，需要更多时间准备模型和优化 Canvas 交互体验。
